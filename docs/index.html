01 · Philosophy — risk-first sizing

Most order tickets ask "how much size / what leverage?" Viridis inverts that. You state the one thing you actually know — how many dollars you're willing to lose — plus your entry and stop. Everything else is derived:

The whole idea risk ($) + entry + stop  ⟶  position size, leverage, and a full OTOCO bracket
  • Position size comes from risk, never from margin or a leverage slider.
  • Leverage is computed so liquidation always sits behind your stop — it is a capital-efficiency knob, not a risk knob.
  • "Risk" means net loss — the size is solved so that the price move to your stop plus round-trip fees equals the dollars you typed.

The result: if you're wrong you lose exactly what you said; if you're right you keep the rest. Leverage changing from 20× to 50× does not change either number — only how much margin is locked.

02 · Architecture & signal flow

Six modules, sharply separated. All exchange I/O is non-blocking; all GUI updates travel through Qt signals.

config.py
.env → typed constants: API keys, fee rates, SL trigger reference, cache & journal settings.
cache_engine.py
Instrument symbology + per-symbol maintenance-margin rate. Paginated pull, cached to disk (24h TTL), O(1) lookup.
trading_core.py
The engine: risk math, OTOCO execution, REST + dual WebSocket state machine, Strat1, journal reconciliation. No GUI.
journal.py
Append-only trade journal, deduped by closing order ID, mirrored from Bybit's closed-PnL. Stats + R-multiples.
main.py
PyQt6 terminal GUI. Owns the SignalBridge, renders trade cards, preview, journal, logs.
test_trade.py
CLI dry-run harness: prints the exact payload, leverage, margin, and fee estimate before optionally firing.

The rule that keeps the GUI responsive: background threads never touch widgets directly. They mutate state under a lock, then hand an immutable snapshot to the main thread via a Qt signal.

03 · The risk math

Position size (fee-aware)

When a long is stopped out you lose the price move and pay fees on the notional — a maker fee on the limit entry and a taker fee on the market stop. Both scale with notional, which balloons as the stop tightens. So the size is solved with the fees inside the denominator:

Fee-aware quantity qty = risk  /  ( |entry − sl|  +  entry·fmaker  +  sl·ftaker ) 
# Worked example — BTC long, entry 74,300 · SL 73,300 · risk $50
fee/unit  = 74300·0.0002 + 73300·0.00055   = 55.18
qty       = 50 / (1000 + 55.18)            = 0.04738  → 0.047 (round down to step)

# Naive sizing (risk/|entry−sl|) would give 0.050 and a TRUE loss of $52.76 — a 5.5%
# overshoot here, ballooning to ~37% on a 0.2% stop. Fee-aware holds the loss at $50.

Safe leverage

Leverage is chosen so the liquidation distance always exceeds the stop distance, with a cushion for the stop order's own fill slippage. The exact isolated-margin liquidation distance for a long is:

Liquidation distance (fraction of entry) liq_dist = ( 1/L − MMR ) / ( 1 − MMR )

The engine uses the slightly conservative approximation 1/L − MMR (dropping the 1−MMR denominator makes it under-estimate the distance — i.e. it errs toward safer, lower leverage), and requires it to clear the stop plus a buffer:

Chosen leverage L = floor( 1 / ( sl_distance + MMR + fee_cushion ) )   capped at the symbol's max 
# BTC: sl_distance = 1000/74300 = 1.346% · MMR = 0.5% · cushion = 0.2%
L = floor( 1 / (0.01346 + 0.005 + 0.002) ) = floor(48.9) = 48×

Why a Mark-Price stop trigger?

Liquidation is always evaluated on Mark Price. If your stop triggered on Last Price, your cushion would be measured between two different rulers — and Last vs Mark diverge exactly during the violent moves where the cushion matters. Triggering the SL on Mark puts it on the same ruler as the liquidation engine, so the stop is guaranteed to fire first. The leverage cushion above is then simply the room for the market stop to fill. Configurable via SL_TRIGGER_BY.

04 · Order execution — the atomic bracket

A single place_order call carries the entry plus its dormant TP/SL, so the entry and its bracket arrive together (OTOCO). With tpslMode="Partial" the TP is a real limit order (maker fee), the SL is a market order (guaranteed exit), and multi-target scale-outs stay possible later.

{
  orderType:   "Limit",   price: entry,   qty: <fee-aware>,
  takeProfit:  <midpoint trigger>,  tpOrderType: "Limit",  tpLimitPrice: tp,
  stopLoss:    sl,           slOrderType: "Market",  slTriggerBy: "MarkPrice",
  tpslMode:    "Partial",   timeInForce: "GTC" (or "PostOnly" — cancel instead of crossing the book)
}

The GUI shows the symbol's live last/bid/ask while you type (click to use as entry) and warns when a limit entry would cross the book — a marketable limit fills immediately as taker, at a different price and fee than the sizing assumed. The optional post-only checkbox turns that warning into an exchange-side guarantee. After the fill, the private execution stream verifies the assumption: a taker entry is flagged in the log with actual vs planned fees. A live position can always be exited instantly via close mkt (reduce-only market order + a sweep that cancels any orphaned TP/SL conditionals).

The midpoint-trigger trick (for a maker TP fill)

The TP trigger is set at the midpoint between entry and target, while the limit price sits at the target. So when price reaches the midpoint, a limit order is placed at the target and rests on the book — by the time price arrives, it fills as a maker, not a taker.

SLmarket exit
entrylimit fill
midpointTP trigger fires → limit rests
TPmaker fill
Long shown. The limit sell is placed at the target the moment price crosses the midpoint, so it's resting (maker) before price gets there.

05 · Concurrency model

This is the most intricate part of the system: four execution contexts cooperate, and exactly one of them is allowed to touch the GUI. Understanding the lanes is the key to understanding the code.

① Qt main thread
  • All widget rendering & reads
  • 30s position-sync timer
  • 500ms panel-flush timer
  • Spawns workers for execute / modify / cancel so it never blocks
② Private WS
  • order_stream → fills, cancels, TP/SL hits
  • position_stream → uPnL, mark, realised PnL
  • execution_stream → per-fill fees, maker/taker truth
  • wallet_stream → live equity
  • Mutates TradeState under the lock
③ Public WS
  • ticker_stream → live mark + last/bid/ask cache
  • Computes live PnL locally
  • Feeds the pre-trade price display
  • Evaluates Strat1 each tick
④ Worker queue
  • Runs REST ops dispatched from the WS threads (e.g. Strat1 amends)
  • Keeps slow REST off the WS callbacks
shared state: self._trades (dict of TradeState)  ·  guarded by a single RLock  ·  every cross-thread update reaches the GUI only as a clone() snapshot over a Qt signal

Two patterns make this safe: (1) snapshots, not shared objects — threads emit clone() copies so the UI never reads an object mid-mutation; (2) optimistic local update + rollback — Strat1 updates its in-memory stop immediately to suppress duplicate triggers, dispatches the REST amend to the worker, and rolls back if the amend fails.

06 · Trade lifecycle

Each trade is a small state machine keyed first by our client order ID, then re-keyed to Bybit's order ID once it's known.

A partial fill that gets its remainder cancelled promotes straight to LIVE at the filled size. A LIVE trade can also exit via close mkt (reduce-only market order, recorded as a Manual close). On boot, sync_existing() rebuilds this machine for any positions or resting orders already on the account — including ones placed elsewhere — and re-arms Strat1 from the risk ledger on trades that were opened with it. The status pill reflects actual WebSocket health (polled every 5s) and degrades visibly to the 30s REST fallback when a stream drops.

07 · Strat1 — the de-risking trail

Strat1 is a deliberately simple heuristic, not a profit-maximiser. Because targets sit near expected resistance, once price has "basically done the job" the goal is to not give it back — so the stop ratchets toward break-even in two phases, measured as progress along the entry → TP journey:

TriggerActionPhase
≥ 75% to TPStop → entry ± (half the original stop distance)1
≥ 90% to TPStop → true break-even (offset past entry to cover round-trip fees, so a stop-out nets ~$0)2

Amendments are rate-limited to one per 5 seconds per symbol, applied optimistically in memory, executed on the worker thread, and rolled back if the exchange rejects them. It never trails into profit — by design; that would be a different strategy.

Strat1 survives restarts. The original SL/TP and the current phase are persisted in the risk ledger at placement and on every successful amend; on boot, sync_existing() matches the position back to its ledger entry (by client order ID for resting entries, by symbol + entry price for live positions) and re-arms the trail at the recorded phase. Trades placed before this persistence existed can't be restored — the app logs a loud warning instead of failing silently.
